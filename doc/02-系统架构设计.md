# 系统架构设计

## 1. 架构目标

系统采用轻量分层架构，目标是让 Streamlit 页面、图像识别模型、热量计算、GPT 建议和 SQLite 数据存储相互解耦，便于开发、测试、展示和后续扩展。

## 2. 总体架构

```text
用户浏览器（PC/手机）
  ↓
Streamlit 交互界面
  ↓
业务服务层
  ├─ 图片上传与校验
  ├─ 模型预测服务
  ├─ 热量计算服务
  ├─ GPT饮食建议服务
  └─ 历史记录服务
  ↓
AI模型层
  └─ PyTorch ResNet18 食物分类模型
  ↓
数据存储层
  ├─ SQLite 食物热量表
  ├─ SQLite 识别历史表
  ├─ SQLite GPT建议日志表
  └─ 本地模型权重文件
```

## 3. 分层说明

| 层级 | 组成 | 职责 |
|---|---|---|
| 表现层 | Streamlit 页面 | 图片上传、表单输入、结果展示、历史表格、图表展示 |
| 业务服务层 | `services/` 模块 | 封装预测、热量计算、数据库、GPT 调用等业务逻辑 |
| AI 模型层 | PyTorch + ResNet18 | 对上传图片进行分类，输出类别和置信度 |
| 数据存储层 | SQLite + 本地文件 | 保存热量表、历史记录、建议日志、模型权重和类别映射 |
| 外部服务层 | OpenAI GPT-5 API | 根据识别结果生成自然语言饮食建议 |

## 4. 核心数据流

### 4.1 食物识别流程

```text
上传图片
  ↓
图片格式校验
  ↓
Pillow/OpenCV 读取图片
  ↓
resize 到 224×224
  ↓
归一化
  ↓
ResNet18 模型推理
  ↓
Softmax 输出概率
  ↓
返回预测类别、置信度、Top-3
```

### 4.2 热量估算流程

```text
预测类别
  ↓
查询 SQLite 食物热量表
  ↓
用户输入重量
  ↓
计算：每100g热量 × 重量 / 100
  ↓
返回估算热量
```

### 4.3 GPT 饮食建议流程

```text
食物名称 + 重量 + 估算热量 + 用户目标
  ↓
构造 Prompt
  ↓
调用 GPT-5 API
  ↓
返回中文饮食建议
  ↓
保存建议日志
```

如果 GPT 调用失败，系统改用本地规则建议，并在日志中记录 `status = fallback`。

## 6. 关键设计原则

- **识别和建议分离**：ResNet18 负责图像分类，GPT-5 负责语言建议，两者职责不混用。
- **估算和称重分离**：图片只用于识别类别，重量由用户输入或默认值提供。
- **页面和业务分离**：Streamlit 页面只负责交互展示，复杂逻辑放入 `services/`。
- **可降级运行**：即使 GPT API 不可用，核心识别和热量估算仍可运行。
- **数据可追踪**：识别历史、热量结果和 GPT 建议都保存到 SQLite。

## 7. 扩展方向

后续可以在不改变主体架构的基础上扩展：

- 增加更多 Food-101 类别。
- 将 ResNet18 替换为 MobileNetV2 或 EfficientNet 做模型对比。
- 增加目标检测模型，支持一张图中识别多个食物。
- 增加用户账户和每日饮食目标。
- 部署到局域网服务器或云服务器。
