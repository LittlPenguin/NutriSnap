# 课程设计报告写作提纲

## 1. 报告题目

**基于 Streamlit 与迁移学习的食物图像识别与热量估算系统**

## 2. 报告结构

根据参考模板，建议报告采用以下结构：

```text
1 系统需求分析
2 系统设计
  2.1 系统架构设计
  2.2 人工智能相关技术原理及模块设计
  2.3 功能模块设计
  2.4 数据存储设计
3 系统实现
  3.1 食物图像识别功能实现
  3.2 热量估算功能实现
  3.3 GPT饮食建议功能实现
  3.4 历史记录与统计功能实现
4 课程设计总结及心得体会
```

## 3. 第 1 章：系统需求分析

### 3.1 可写内容

- 项目背景：健康饮食、热量记录、食物识别。
- 系统主题：基于图片识别食物类别并估算热量。
- 用户角色：普通用户、系统管理员/开发者。
- 功能需求：上传、识别、估算、建议、历史、统计。
- 非功能需求：易用性、稳定性、移动端兼容性、安全性。

### 3.2 建议表述

```text
本系统面向日常饮食健康管理场景，用户通过上传食物图片快速获得食物类别识别结果，并结合输入重量计算估算热量。系统进一步调用 GPT-5 生成饮食建议，帮助用户更直观地了解本次饮食摄入情况。
```

### 3.3 配图建议

- 用例图。
- 用户操作流程图。

## 4. 第 2 章：系统设计

## 4.1 系统架构设计

写作重点：

- Streamlit 表现层。
- 业务服务层。
- PyTorch 模型层。
- SQLite 数据层。
- GPT-5 服务层。

建议架构图：

```text
用户浏览器
  ↓
Streamlit 页面
  ↓
业务服务模块
  ↓
PyTorch ResNet18 / SQLite / GPT-5
```

## 4.2 人工智能相关技术原理及模块设计

这是文档重点部分。

建议写：

- CNN 卷积神经网络原理。
- ResNet18 残差连接原理。
- 迁移学习原理。
- Softmax 分类概率。
- GPT-5 饮食建议模块的作用。

注意区分：

- ResNet18 负责图像识别。
- GPT-5 负责饮食建议。
- 热量估算是规则计算，不是模型直接估计。

## 4.4 界面与交互设计

建议增加一小节说明 UI 设计稿与实现的对应关系：

- UI 采用“NutriSnap 轻食记录”风格。
- 移动端包含 5 个独立页面：食物识别、历史记录、食物热量表、统计分析、系统说明。
- PC 端包含 5 个宽屏页面：食物识别工作台、历史记录、食物热量表、统计分析、系统说明。
- 系统说明页展示项目简介、使用步骤、功能边界和免责声明，不展示技术栈；不要在系统说明页展示技术栈。

- 示例数据统一使用披萨、150g、266 kcal / 100g、399 kcal。

## 5. 第 3 章：系统实现

## 5.2 热量估算功能实现

可写内容：

- 食物热量表。
- 重量输入。
- 热量计算公式。
- 页面结果展示。

公式：

```text
估算热量 = 每100g热量 × 用户输入重量 / 100
```

需要说明：

```text
本系统不根据图片直接估计食物重量，用户需要输入或选择食物重量，因此热量结果为辅助估算值。
```

重点说明：

- 不上传图片给 GPT。

- GPT 建议不作为医学诊断。

## 6. 第 4 章：课程设计总结及心得体会

建议包含：

- 项目完成情况。
- 技术收获。
- 系统优点。
- 系统不足。
- 改进方向。
- 自我评价和评分。

不足可以写：

- 一期只支持 12 类食物。
- 图片角度、光照和遮挡会影响识别准确率。
- 热量估算依赖用户输入重量。
- GPT 建议受 API 可用性影响。

改进方向可以写：

- 增加更多食物类别。
- 引入目标检测识别一图多食物。
- 增加用户账户和每日摄入目标。
- 引入更专业的营养数据库。
- 增加移动端封装或云端部署。

## 8. 写作注意事项

- 全文统一使用题目：基于 Streamlit 与迁移学习的食物图像识别与热量估算系统。
- 不要写系统能“精准计算热量”，应写“估算热量”或“辅助参考”。
- 不要写 GPT-5 负责图片识别，图片识别由 ResNet18 完成。
- 不要写建议具有医疗诊断作用。
- 技术路线应始终保持：Streamlit + PyTorch ResNet18 + SQLite + GPT-5。
- 报告中的 UI 截图应与 `end/opendesign-nutrisnap/` 最新设计稿一致。
- 不要在系统说明页截图中展示技术栈；技术栈放在报告文字或答辩口播中说明。
