# 界面与交互设计

## 1. 设计目标

系统使用 Streamlit 构建交互界面，同时适配 PC 端浏览器和移动端浏览器。界面应突出核心流程：上传图片、识别食物、输入重量、查看估算热量、生成 GPT 饮食建议、查看历史记录和统计趋势。

UI 设计以 Open Design 最新稿为准：

- Open Design 项目：`NutriSnap 轻食记录`
- 项目 ID：`0fbcf412-3ba4-4dbf-b769-d965b9ef7e69`
- 本地归档目录：`end/opendesign-nutrisnap/`
- 移动端：5 个独立页面。
- PC 端：5 个宽屏页面。
- 系统说明页不展示技术栈。

## 2. 设计风格

设计风格采用“移动端轻食记录”：

- 清爽、亲和、轻量、低压力。
- 像日常饮食记录工具，不像医疗诊断系统。
- 不做外卖平台风格。
- 不做复杂后台 Dashboard 风格。
- 热量结果统一写作“估算热量”，并说明仅供饮食记录参考。

配色基准：

| 用途 | 色值 |
|---|---|
| 页面背景 | `#F7FAF6` |
| 主色 | `#5CA878` |
| 强调色 | `#F4A261` |
| 主文本 | `#1F2933` |
| 次级文本 | `#6B7280` |
| 卡片背景 | `#FFFFFF` |
| 边框 | `#E5E7EB` |
| GPT 建议背景 | `#EEF8F1` |
| 降级提示背景 | `#FFF4E6` |

组件规则：

- 普通卡片圆角统一 8px。
- 使用白底卡片、浅边框和轻阴影。
- 热量数字使用柔和橙色突出。
- GPT 建议使用浅绿色卡片。
- GPT 失败或本地规则建议使用柔和橙色提示卡片。

## 3. 页面结构

Streamlit 实现时建议使用五个主页面或五个标签页：

```python
st.tabs(["食物识别", "历史记录", "热量表", "统计分析", "系统说明"])
```

页面职责：

| 页面 | 职责 |
|---|---|
| 食物识别 | 上传图片、预览图片、识别类别、展示 Top-3、输入重量、估算热量、生成 GPT 建议 |
| 历史记录 | 展示识别历史、重量、估算热量、置信度、建议摘要和时间 |
| 热量表 | 展示 Food-101 子集支持类别、中文名、英文类名、默认份量、每 100g 热量 |
| 统计分析 | 展示累计记录、累计热量、今日热量、近 7 日趋势和常见食物排行 |
| 系统说明 | 展示项目简介、使用步骤、模型与数据边界、GPT-5 功能边界、免责声明 |

注意：系统说明页不展示技术栈卡片或技术栈信息块。技术栈可以在项目总览、课程报告、演示讲解中介绍。

## 4. 移动端 5 个独立页面

移动端按 390 x 844 画布设计，模拟手机浏览器访问体验。每个页面都使用单列布局和统一底部导航。

底部导航：

```text
识别 / 历史 / 热量表 / 统计 / 说明
```

### 4.1 移动端食物识别页

页面包含：

- 顶部标题：`NutriSnap`，副文案 `今天也记一餐`。
- 今日记录概览：今日已记录次数和今日估算热量。
- 图片上传卡片：未上传状态、拍照 / 相册入口、图片预览。
- 开始识别按钮。
- 识别结果卡片：食物类别、置信度、Top-3。
- 份量输入：默认 `150g`，用户可修改。
- 估算热量卡片：每 100g 热量和总热量。
- 用户目标选择：普通饮食、减脂、增肌、控糖。
- GPT 饮食建议卡片。

示例数据：

```text
食物：披萨
置信度：93.0%
Top-3：披萨 93%、汉堡 4%、芝士蛋糕 1%
重量：150g
每 100g 热量：266 kcal / 100g
估算热量：399 kcal
```

### 4.2 移动端历史记录页

页面包含：

- 今日记录次数。
- 今日估算热量。
- 历史记录卡片列表。
- 每条记录显示食物名、重量、估算热量、置信度、时间、GPT 建议摘要。
- GPT 失败时显示“本地规则建议”降级提示。

移动端不展示过宽表格，优先使用紧凑卡片。

### 4.3 移动端食物热量表页

页面包含：

- 搜索框。
- 分类筛选：全部、主食、快餐、甜点、肉类、沙拉。
- 食物热量卡片列表。
- 每条展示中文名、英文类名或类别、默认份量、每 100g 热量。

示例：

```text
披萨
默认份量 150g
266 kcal / 100g
```

### 4.4 移动端统计分析页

页面包含：

- 累计记录。
- 累计热量。
- 今日热量。
- 近 7 日估算热量柱状图。
- 常见食物排行。

图表保持简洁，以绿色和柔和橙色为主，不做复杂仪表盘。

### 4.5 移动端系统说明页

页面包含：

- 项目简介。
- 使用步骤。
- 注意事项。
- 免责声明。

页面不展示技术栈。技术栈内容只在项目文档、报告或演示口播中说明。

## 5. PC 端 5 个宽屏页面

PC 端按 1440 x 900 画布设计，使用顶部导航和宽屏内容布局。PC 端不是只有首页，而是完整 5 个页面。

顶部导航：

```text
食物识别 / 历史记录 / 热量表 / 统计分析 / 系统说明
```

### 5.1 PC 食物识别工作台

布局：

- 左侧：图片上传、图片预览、开始识别、重新上传。
- 右侧：识别结果、置信度、重量、每 100g 热量、Top-3、估算热量、用户目标、GPT 饮食建议。

适合使用 Streamlit `st.columns()` 实现左右布局。

### 5.2 PC 历史记录页

布局：

- 顶部指标：今日记录、今日估算热量、累计记录、平均置信度。
- 历史记录表格：食物、重量、估算热量、置信度、建议摘要 / 时间。
- 下方卡片：GPT 饮食建议摘要、本地规则建议。

### 5.3 PC 食物热量表页

布局：

- 查询与筛选区。
- 热量表格：中文名称、Food-101 类名、类别、默认份量、每 100g 热量。
- 热量表边界说明。

### 5.4 PC 统计分析页

布局：

- 顶部指标：累计记录、累计热量、今日热量、常见食物。
- 近 7 日估算热量柱状图。
- 常见食物排行。
- 结果边界说明。

### 5.5 PC 系统说明页

页面包含：

- 项目简介。
- 使用步骤。
- 模型与数据边界。
- GPT-5 功能边界。
- 免责声明。

页面不展示技术栈卡片或技术栈信息块。

## 6. 交互流程

食物识别页核心流程：

1. 用户上传图片。
2. 系统显示图片预览。
3. 用户点击“开始识别”。
4. 系统显示加载状态。
5. 系统输出预测类别、置信度和 Top-3。
6. 系统读取默认重量。
7. 用户可修改重量。
8. 系统根据每 100g 热量计算估算热量。
9. 用户选择饮食目标。
10. 系统生成 GPT 饮食建议。
11. 如果 GPT 失败，系统显示本地规则建议。
12. 系统保存历史记录。

## 7. 状态覆盖

设计稿和后续实现至少覆盖以下状态：

- 未上传图片。
- 已上传图片预览。
- 正在识别。
- 识别完成。
- GPT 建议生成中。
- GPT 建议完成。
- GPT API 失败后的本地规则建议。
- 历史记录正常显示。
- 移动端单列布局。
- PC 端宽屏布局。

## 8. 文案边界

允许使用：

- 估算热量。
- 仅供饮食记录参考。
- 不作为医学或营养诊断。
- 本地规则建议。

禁止使用：

- 精准计算热量。
- 医学诊断。
- 治疗建议。
- 仅凭图片精确估重。

## 9. Streamlit 落地建议

- 使用 `st.set_page_config(layout="wide")` 支持 PC 宽布局。
- 使用 `st.tabs()` 或 Streamlit 多页面实现五个功能页。
- PC 端食物识别页使用 `st.columns()` 形成左右布局。
- 移动端依靠 Streamlit 自适应和单列堆叠，避免宽表格直接暴露。
- 历史记录在 PC 端可用表格，在移动端优先展示关键字段。
- 可使用少量 CSS 对卡片、按钮、颜色和间距进行轻量定制。
