# 需求分析

## 1. 需求概述

本系统服务于日常饮食记录与热量参考场景。用户在 PC 或手机浏览器中打开系统，上传食物图片，系统识别食物类别，用户输入食物重量后获得估算热量，并可查看 GPT-5 生成的饮食建议和历史记录。

系统的核心目标不是替代专业营养工具，而是为普通用户提供低门槛、可视化、可演示的热量估算辅助工具。

## 2. 用户角色

| 角色 | 说明 | 主要操作 |
|---|---|---|
| 普通用户 | 使用系统记录饮食和查看建议的人 | 上传图片、查看识别结果、输入重量、生成建议、查看历史 |
| 系统管理员/开发者 | 维护系统数据、模型和配置的人 | 准备数据集、训练模型、维护热量表、配置 GPT API |
| 答辩演示者 | 课程项目展示人员 | 演示上传、识别、估算、建议、历史和统计流程 |

## 3. 典型使用场景

### 3.1 PC 端演示场景

1. 用户在电脑浏览器打开 Streamlit 应用。
2. 在“食物识别”页上传一张食物图片。
3. 系统显示图片预览。
4. 用户点击“开始识别”。
5. 系统显示预测类别、置信度和 Top-3 预测结果。
6. 用户输入食物重量。
7. 系统计算估算热量。
8. 用户选择目标，例如“减脂”。
9. 系统调用 GPT-5 生成饮食建议。
10. 系统保存识别历史。

### 3.2 移动端使用场景

1. 手机和电脑连接同一 Wi-Fi。
2. 用户用手机浏览器访问 Streamlit 地址。
3. 用户通过手机相册或拍照上传食物图片。
4. 页面以单列布局展示图片、识别结果、重量输入框、热量结果和建议。
5. 用户查看历史记录和统计图表。

## 4. 功能需求

| 编号 | 需求 | 说明 | 优先级 |
|---|---|---|---|
| FR-01 | 图片上传与预览 | 支持 jpg、jpeg、png，上传后展示预览 | 必须 |
| FR-02 | 图片格式校验 | 非图片文件或无法读取的图片应提示错误 | 必须 |
| FR-03 | 食物类别识别 | 使用 ResNet18 输出预测类别和置信度 | 必须 |
| FR-04 | Top-3 展示 | 显示前三个可能类别和对应置信度 | 必须 |
| FR-05 | 重量输入 | 用户可输入克数，默认 100g | 必须 |
| FR-06 | 热量估算 | 根据每 100g 热量和重量计算总热量 | 必须 |
| FR-07 | GPT 饮食建议 | 根据识别结果、热量、目标生成建议 | 必须 |
| FR-08 | 本地建议降级 | GPT API 不可用时使用本地规则建议 | 必须 |
| FR-09 | 历史记录保存 | 保存识别结果、重量、热量、建议和时间 | 必须 |
| FR-10 | 历史记录查询 | 按时间倒序展示历史记录 | 必须 |
| FR-11 | 食物热量表 | 展示支持类别的热量数据 | 必须 |
| FR-12 | 统计分析 | 展示总热量和按日期汇总图表 | 应有 |
| FR-13 | 系统说明 | 展示项目介绍、使用方法和限制说明 | 应有 |

## 5. 非功能需求

| 类型 | 要求 |
|---|---|
| 性能 | 单张图片本地模型预测建议控制在 3 秒以内；GPT 建议单独显示加载状态 |
| 兼容性 | 支持 PC 浏览器和移动端浏览器访问 |
| 可用性 | 上传、识别、输入重量、生成建议流程必须清晰 |
| 稳定性 | 模型文件缺失、热量数据缺失、GPT 调用失败时给出明确提示 |
| 可维护性 | 模型预测、热量计算、数据库、GPT 服务拆分为独立模块 |
| 安全性 | OpenAI API Key 使用 `.env` 配置，不写入文档或公开源码 |
| 合规性 | 明确系统输出为饮食参考，不构成医学或营养诊断 |

## 6. 用例说明

| 用例 | 参与角色 | 前置条件 | 基本流程 | 结果 |
|---|---|---|---|---|
| 上传并识别食物 | 普通用户 | 系统已启动，模型文件存在 | 上传图片，点击识别 | 显示类别、置信度、Top-3 |
| 估算食物热量 | 普通用户 | 已获得识别类别 | 输入重量，点击计算 | 显示估算热量 |
| 生成饮食建议 | 普通用户 | 已计算热量 | 选择目标，调用 GPT | 显示建议文本 |
| 查看历史记录 | 普通用户 | 数据库已有历史 | 打开历史记录页 | 显示历史列表 |
| 查看统计分析 | 普通用户 | 数据库已有历史 | 打开统计分析页 | 显示热量统计图 |
| 维护热量表 | 开发者 | 需要更新支持类别 | 修改初始化 CSV 或数据库 | 更新类别和热量数据 |

## 7. 验收标准

- 用户能完成从上传图片到获得识别结果、估算热量和饮食建议的完整流程。
- 支持 PC 和移动端浏览器访问，移动端页面不出现严重横向溢出。
- GPT API 不可用时，系统仍能给出本地规则建议。
- 历史记录和统计分析能反映实际识别记录。
- 文档和演示中明确说明热量估算的局限性。
